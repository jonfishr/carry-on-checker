🤖 Automated Data Monitoring Architecture

Keeping airline rules and luggage data up-to-date automatically

🎯 System Overview

This architecture monitors airline websites and luggage manufacturers for changes to carry-on policies and product dimensions, then flags changes for admin review before automatically updating the website.

Key Design Principles:
  • Human-in-the-Loop: All changes require admin verification before going live
  • Cost-Conscious: Leverages free tiers and serverless technology
  • Reliable: Multiple fallback strategies for data collection
  • Scalable: Can monitor hundreds of sources without infrastructure changes

🏗️ Architecture Components

1

Web Scraping Service

Scheduled jobs that check airline websites and manufacturer pages for changes to dimensions and policies.

2

Change Detection

Compares new data against existing database, flags differences, and calculates confidence scores.

3

Admin Dashboard

Review interface where you approve/reject changes, edit details, and manage monitoring sources.

4

Auto-Deployment

Approved changes update JSON files and trigger automatic website deployment via GitHub.

🛠️ Technical Stack (Recommended)

⏰ Scheduler

GitHub Actions (Free)

Run monitoring scripts on a schedule (daily/weekly). 2,000 free minutes/month is more than enough.

🔍 Scraping Engine

Python + Playwright

Headless browser automation for JavaScript-heavy sites. More reliable than basic HTTP requests.

🧠 LLM Processing

Claude API (Haiku)

Extract structured data from scraped HTML. Haiku is fast and cheap ($0.25 per million tokens).

💾 Database

Airtable (Free Tier)

Acts as your data store, admin interface, and approval workflow. 1,200 records free is plenty.

🔔 Notifications

Email via Resend (Free)

100 emails/day free. Sends alerts when changes are detected for your review.

🚀 Deployment

GitHub Actions

Automatically deploys approved changes to Cloudflare Pages when you update the JSON files.

⚠️ Important: Verification Steps

When you see a pending change:

  1. Click the source URL to verify the information
  2. Check the screenshot captured during scraping
  3. Review the confidence score (>80% is usually reliable)
  4. If correct, click "Approve" to update the website
  5. If wrong, click "Reject" and the system will learn from it

When you approve a change in Airtable, here's what happens automatically:

# GitHub Actions workflow (.github/workflows/deploy.yml)

name: Deploy Updates
on:
  # Trigger when Airtable sends webhook OR on manual dispatch
  repository_dispatch:
    types: [airtable-approval]
  workflow_dispatch:

jobs:
  update-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v3
      
      - name: Fetch approved changes from Airtable
        run: python scripts/fetch_approved_changes.py
        env:
          AIRTABLE_API_KEY: ${{ secrets.AIRTABLE_API_KEY }}
      
      - name: Update JSON files
        run: python scripts/update_json.py
      
      - name: Commit changes
        run: |
          git config user.name "Automation Bot"
          git config user.email "bot@example.com"
          git add data/*.json
          git commit -m "Auto-update: Approved dimension changes"
          git push
      
      - name: Trigger Cloudflare Pages deployment
        # Cloudflare Pages auto-deploys on git push
        run: echo "Deployment triggered!"

🔍 Data Extraction Strategy

Challenge: Airline websites are all different

Each airline formats their baggage policy differently. Here's how we handle it:

Solution 1: Claude Vision (Recommended)

# Take a screenshot of the policy page and use Claude to extract data
def extract_with_vision(screenshot_path):
    response = anthropic.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1000,
        messages=[{
            "role": "user",
            "content": [
                {
                    "type": "image",
                    "source": {
                        "type": "base64",
                        "media_type": "image/png",
                        "data": base64.b64encode(open(screenshot_path, 'rb').read()).decode()
                    }
                },
                {
                    "type": "text",
                    "text": """
                    Extract the carry-on baggage dimensions from this airline policy page.
                    Look for maximum allowed dimensions (height x width x depth).
                    
                    Return ONLY JSON in this exact format:
                    {
                        "height": 55,
                        "width": 40,
                        "depth": 23,
                        "unit": "cm",
                        "confidence": 95
                    }
                    """
                }
            ]
        }]
    )
    return json.loads(response.content[0].text)

Solution 2: HTML Parsing (Backup)

# Parse the raw HTML with Claude
def extract_from_html(html_content):
    response = anthropic.messages.create(
        model="claude-haiku-4-20250514",  # Cheaper for text processing
        max_tokens=500,
        messages=[{
            "role": "user",
            "content": f"""
            Extract carry-on dimensions from this HTML.
            Focus on sections about 'carry-on', 'cabin baggage', or 'hand luggage'.
            
            HTML:
            {html_content[:10000]}  # Truncate to save tokens
            
            Return JSON: {{"height": X, "width": Y, "depth": Z, "unit": "cm", "confidence": N}}
            """
        }]
    )
    return json.loads(response.content[0].text)
Why this works:
  • Claude understands context (e.g., "55x40x23cm" vs "Linear: 115cm")
  • Can handle different formats (tables, lists, paragraphs)
  • Returns confidence score so you know when to double-check
  • Works across multiple languages (for international airlines)

🛡️ Error Handling & Quality Control

Built-in Safety Mechanisms:

1. Confidence Scoring

Claude returns a confidence level (0-100%). Only high-confidence changes (>80%) are flagged for quick review.

2. Screenshot Evidence

Every change includes a screenshot of the source, so you can visually verify before approving.

3. Sanity Checks

Automatic validation (e.g., height must be 40-60cm, width 30-50cm) prevents obviously wrong data.

4. Change History

Airtable keeps full history. You can rollback if something goes wrong.

5. Duplicate Detection

System won't flag the same change twice, reducing notification spam.

6. Rate Limiting

Respects robots.txt and adds delays between requests to be a good citizen.

# Example validation logic
def validate_dimensions(height, width, depth, unit="cm"):
    # Convert to cm if needed
    if unit == "in":
        height, width, depth = height * 2.54, width * 2.54, depth * 2.54
    
    # Sanity checks
    if not (40 <= height <= 65):
        return False, "Height out of reasonable range"
    if not (25 <= width <= 55):
        return False, "Width out of reasonable range"
    if not (15 <= depth <= 35):
        return False, "Depth out of reasonable range"
    
    return True, "Valid"

🎛️ Alternative Approach: Community-Driven Updates

If full automation feels too complex initially, consider a hybrid approach:

Option A: User-Submitted Updates

  • Add a "Report a Change" button to your website
  • Users submit airline/luggage updates with source links
  • You review and approve in Airtable
  • Pro: No scraping infrastructure needed, community engagement
  • Con: Relies on user participation

Option B: Semi-Automated (Best of Both)

  • Run automated checks monthly for major airlines
  • Accept community submissions for everything else
  • All changes still require your approval
  • Pro: Balance of automation and community input
  • Con: Still requires some monitoring infrastructure
